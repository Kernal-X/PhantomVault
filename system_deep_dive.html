Agentic Security System

Full technical deep dive: architecture, LangGraph workflow, ML integration, LLM planning, deception pipeline, runtime behavior, demos, and known limitations.

LangGraph Orchestration Host Monitoring ML Risk Scoring LLM Analysis Deception Strategy File-Focused Response
Core idea: the system monitors host telemetry, detects suspicious behavior, estimates risk with ML, reasons about attacker intent with LLMs, plans deception, deploys decoys, and conditionally serves fake content through its interception flow.

1. Executive Summary

The project is a layered security pipeline built around one explicit orchestration graph. It monitors process, file, and network activity, derives enriched behavioral features, filters noise, scores suspicious events, routes events into specialized ML models, aggregates risk, and only then escalates into LLM-based analysis and deception planning.

The strongest and most complete part of the system is the file-side deception path: strategy planning, decoy deployment metadata, file interception decisions, and realistic fake content generation. LangGraph is used as the control plane that ties these modules together into a single, testable workflow.

main.py -> SystemAgent -> LangGraphSecurityPipeline -> monitor -> enrich -> filter -> score -> emit_alerts -> analysis -> strategy -> deployment -> interception

Primary runtime files

Main subsystems

6. Detection and ML Integration

LangGraph does not directly run the models itself. Instead, the graph reaches the ML layer through the emit_alerts node, which calls the logger. The logger builds normalized ML payloads, routes them to the correct model, and sends outputs into the streaming aggregator.

flowchart LR A["score_events node"] --> B["emit_alerts node"] B --> C["SOCLogger.emit()"] C --> D["Build normalized ML payload"] D --> E["ModelRouter.route()"] E --> F1["File Model"] E --> F2["Process Model"] E --> F3["Network Model"] F1 --> G["StreamingAggregator.add_event()"] F2 --> G F3 --> G G --> H["alert_records + risk_score"] H --> I{"Aggregated alert?"} I -->|"No"| J["End Graph"] I -->|"Yes"| K["analysis node"]

Why this design matters

  • Rule-based detection catches suspicious patterns quickly.
  • Specialized models score different event categories differently.
  • The aggregator prevents single weak anomalies from always escalating into expensive LLM reasoning.

7. Analysis Agent

The analysis agent takes aggregated suspicious events and infers attacker intent, attack stage, confidence, and reasoning. It acts as the semantic bridge between raw risk and strategic response.

Inputs

  • risk_score
  • aggregated suspicious events

Outputs

  • intent
  • attack_stage
  • confidence
  • reasoning

Typical intent examples include data_exfiltration, lateral_movement, and reconnaissance. Typical stage values include collection, exfiltration, and credential_access.

8. Strategy Agent

The strategy agent is the planning layer. It takes analysis output and converts it into a structured deception blueprint. It does not create files directly; it decides what should exist, where it should be placed, how much attacker interaction should be allowed, and what should be monitored.

flowchart TD A["analysis input"] --> B["determine safe staging root"] B --> C["build deterministic hints"] C --> D["build strict JSON-only prompt"] D --> E["call strategy LLM"] E --> F["parse JSON"] F --> G["apply deterministic overrides"] G --> H["normalize enums and repair fields"] H --> I["enforce safety and decoy tags"] I --> J["trim plan to policy limits"] J --> K["validate final shape"] K -->|"valid"| L["return strategy + strategy_meta"] K -->|"invalid"| M["build deterministic fallback strategy"] M --> L

Strategy module files

FileRole
agents/strategy/strategy_agent.pyMain controller for the planning pipeline.

Key design decision: the LLM is allowed to propose a plan, but deterministic policy decides the final safe output.

9. Deployment and Interception

Deployment converts the strategy blueprint into operational metadata. Interception consumes that metadata plus attacker analysis to decide whether a file request should return real content, blended content, or fake content.

flowchart LR A["strategy"] --> B["DeploymentManager.deploy()"] B --> C["decoy_registry"] B --> D["global_context"] B --> E["interception_rules"] C --> F["InterceptionLayer.handle()"] D --> F E --> F G["analysis"] --> F H["request_path"] --> F F --> I{"decision_engine"} I -->|"real"| J["read real file"] I -->|"partial"| K["blend real + generated fake"] I -->|"fake"| L["GenerationAgent.generate()"]

The most mature response surface in the project is the file side. The deployment registry and interception logic are designed mainly around decoy file artifacts.

10. SystemAgent Runtime

SystemAgent is the long-running runtime controller. It is responsible for continuously invoking the LangGraph workflow and printing live incident summaries.

Responsibilities

  • load monitoring configuration
  • configure watched file paths
  • construct LangGraphSecurityPipeline
  • run repeated monitoring cycles
  • print human-readable cycle reports

Live cycle reporting

Each interesting cycle can report:

  • how many events were seen
  • event type distribution
  • file paths observed
  • suspicious file targets
  • aggregated risk score
  • analysis summary
  • deployed decoys

13. Strengths

Explicit orchestration

LangGraph makes the workflow visible, conditional, and state-driven instead of implicit.

Layered reasoning

Rules, ML, and LLMs each serve distinct roles rather than competing with each other.

Strong file response path

The project demonstrates coherent file-focused deception from planning to fake content generation.

14. Limitations and Loopholes

Important: this system is not a kernel-level, tamper-proof, full-OS interception product.
  • If an attacker gains complete administrative or kernel-level control of the OS, a user-space orchestration system cannot guarantee persistence.
  • The strongest response path is file-based deception. Other response surfaces are less complete.
  • The interception flow is controlled by application logic, not by a Windows minifilter or universal filesystem hook.
  • The repository still contains an older src/ scaffold, which is not the real runtime path.
  • LLM quality and availability affect the quality of analysis and strategy generation, although safe fallbacks reduce that risk.

Best way to answer this in defense

If asked how the system persists after full OS compromise, the correct answer is:

“If the attacker has full OS control, a user-space system cannot be assumed fully trustworthy. The value of this project is early detection, deception, and visibility before that point, not guaranteed survival under total host compromise.”

15. Future Improvements

  • physically materialize deployed decoys into configured real watch folders by default
  • introduce protected remote logging and remote orchestration
  • expand deception beyond files into stronger network and endpoint surfaces
  • add kernel-level or gateway-level interception if full transparent file substitution is required
  • improve packaging and documentation around the active runtime versus legacy scaffolding

16. Final Summary

This system is best described as a LangGraph-orchestrated host monitoring and deception platform. Its main contribution is not a single detector or single agent, but the way it unifies telemetry collection, risk estimation, semantic reasoning, deception planning, deployment, and controlled response into one coherent workflow.

In practical terms, it detects suspicious behavior, estimates attacker risk, reasons about likely intent, plans targeted decoys, and can serve believable fake artifacts in a file-focused defense scenario. That is the full system story.
